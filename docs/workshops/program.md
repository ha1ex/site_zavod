# Воркшоп «Вы сможете создавать программные продукты» + воркшоп-разбор

Гайд ведущего. Текстовая версия — источник истины для презентации
[`program.html`](./program.html). Аудитория: нетехническая маркетинг-команда
(писатели, редакторы, маркетологи, дизайнеры).

## Формат

**Один большой воркшоп (~3 часа)** — весь путь за одну сессию: GitHub → лендинг →
деплой на Vercel. Затем **неделя практики** (домашка) и **воркшоп-разбор** —
полностью прикладной: разбираем только то, что реально произошло у команды.

```
ДЗ-0 (подготовка) → Воркшоп (весь путь) → неделя практики (ДЗ-1) → Воркшоп-разбор
```

## Цель

Показать команде, что они **сами могут создавать программные продукты**: с помощью
нейронок делать себе приложения и инструменты, вести их в GitHub и публиковать в
продакшен на Vercel. Лендинги — первый практический полигон.

Сквозная цепочка, которую осваивает каждый:

```
Идея → Нейронка пишет код → GitHub хранит версии → Vercel публикует → Работающий продукт
```

Три навыка: **нейронки для кодинга** · **GitHub** · **деплой на Vercel**.

## Главное правило работы

Команда **не печатает команды и не пишет код**. Открывает проект в Claude Code и
говорит с агентом по-русски: «покажи лендинги в браузере», «поменяй заголовок и
сохрани в main», «задеплой на Vercel». Терминологию узнают по ходу — из ответов агента.

---

## Воркшоп (~3 часа) — план

### 1. Подключаем GitHub (~30 мин)
Аккаунт → доступ → копируем проект себе (Conductor / Claude Code). У каждого на
экране работающий проект. Понятия: **GitHub = Word с историей версий**, внутри лежит
код — правила, по которым нейронка собирает результат.

### 2. Первое изменение (~20 мин)
«Поменяй заголовок на этом лендинге и сохрани в main» → «покажи в браузере, как
теперь выглядит» → смотрим историю на GitHub, учимся откатывать.
**Почему одна ветка main:** ветки нужны большим командам разработчиков; нам важнее
скорость и простота — одна общая версия, история страхует.

### 3. Собираем лендинг (~40 мин)
Конвейер: запрос на входе — лендинг на выходе, между ними шаги строго по порядку:

```
Запрос (бриф) → 1 Разбор задачи → 2 Макет → 3 Блоки → 4 Тексты → 5 Проверки → Лендинг
```

Каждый собирает лендинг по готовому ТЗ: «собери лендинг по этому ТЗ и открой в браузере».
Фраза для понимания: «проведи экскурсию по конвейеру — что на каждом шаге и где правила».

**Живой справочник конвейера.** В проект встроен read-only справочник для маркетинга
(«Как устроен конвейер»): 7 этапов (Фабрика ТЗ → Маршрутизация → Быстрый/Поэтапный
конвейер → Иллюстрации → Визуальное ревью → Согласование → Передача в разработку)
+ 3 раздела (Брендовый канон, Валидаторы качества, База знаний). Каждый этап расписан:
назначение, вход → выход, шаги, правила с бейджами HARD/SOFT. Фраза команде:
«Подними справочник конвейера и открой его в браузере» → откроется
`localhost:3000/pipeline`. Показать на воркшопе и приучить ходить туда за ответами.

### 4. Чиним «не очень» (~40 мин)
**Плохой результат — диагноз, а не приговор.** Брак указывает на шаг конвейера:
- не та иллюстрация → шаг 3 «Блоки» → правило «домен → блоки»;
- текст со штампом → шаг 4 «Тексты» → редполитика;
- не та структура → шаги 1–2 «Разбор / Макет» → бриф или макеты.

Чиним правило шага (не лендинг руками) по **универсальному промпту**:

> «Вот скриншот [картинка]. Проблема: [что не так]. Как должно быть: [как надо].
> Поменяй правило на шаге [название шага], чтобы такого больше не было.
> Перегенерируй и покажи. Если ок — сохрани в main.»

### 5. Деплоим на Vercel (~30 мин)

```
Проект на GitHub → Подключили Vercel (один раз) → Деплой (ссылка в интернете)
→ Правка → сохранили в main → сайт обновился сам
```

«Задеплой этот проект на Vercel и дай ссылку» → меняем текст → сохраняем в main →
через минуту прод обновился. Это и есть продакшен.

### Итог воркшопа
У каждого: проект на GitHub, собранный лендинг и **публичная ссылка** на него.

---

## Домашка (неделя): 2 ТЗ → 2 лендинга → улучшение → деплой

Каждый берёт **два ТЗ** и делает **два лендинга**. Главное — цикл улучшения:

```
Сделал лендинг → Нашёл, что не нравится → Предложил решение (какое правило какого шага)
→ Внёс и перегенерил (стало лучше? если нет — ещё итерация) → Задеплоил на Vercel
```

**Часть 2 домашки — правила передачи в разработку** (1–2 ответственных на команду):
сходить к разработчикам → получить **Markdown-документ** с техническими требованиями
к передаче лендинга → отдать агенту («добавь эти правила в репозиторий как правила
передачи в разработку») → правила становятся частью этапа handoff → проверить промптом:
«Лендинг готов — упакуй его для команды разработки по правилам передачи из репозитория».

Принести на разбор: **2 ссылки на задеплоенные лендинги** + список «что чинил и что
поменял в правилах» + **правила передачи в репозитории и один упакованный по ним
лендинг**. Подробно: [`homework/hw1.md`](./homework/hw1.md).

## Воркшоп-разбор (через неделю) — прикладной

Никакой теории — только то, что реально произошло у команды:

1. **Ваши лендинги и деплои** — каждый показывает свои ссылки: что получилось,
   что нет, как прошёл первый деплой на Vercel.
2. **Ошибки и находки** — типовые проблемы недели: к какому шагу конвейера относятся,
   как чинили, что поменяли в правилах; разбор апдейтов репозитория (кто что менял,
   что конфликтует — как не превращать общий проект в хаос).
3. **Как работать дальше** — кто отвечает за какие правила (ответственные за этапы
   конвейера), путь к своим мини-продуктам.

Точная программа разбора собирается из принесённого материала — чем больше реальных
граблей, тем полезнее.

## Роудмеп: конвейер статей (следующая большая задача)

**Контекст:** у команды есть скилл для статей, но нет автоматизированного конвейера —
как у лендингов. **Цель задачи: после внедрения конвейера команда выпускает статей
в 2 раза больше.** Сам конвейер строим на следующих этапах; сейчас в работу берётся
**research** (один ответственный, договариваемся на воркшопе-разборе):

1. **Описать бизнес-процесс** — как статьи рождаются сейчас: от идеи до публикации,
   где узкие места.
2. **Исследовать рынок** — открытые репозитории с готовыми конвейерами создания
   контента для сайтов.
3. **Собрать лучшие практики** — несколько источников, что реально работает.
4. **Подготовить материалы** — всё, что нужно, чтобы собрать свой конвейер статей.

Промпт для старта: «Проведи research: лучшие открытые репозитории с конвейерами
создания контента — сравни подходы и собери лучшие практики».

## Границы

Статьи **пока не реализованы** — конвейер делает только лендинги. Конвейер статей —
по роудмепу выше: сначала research, потом сборка.

## Шпаргалка ведущего (НЕ показывать команде)

Что происходит «под капотом», если агент застрял и нужно помочь руками:

| Человеческая фраза | Что реально делает агент |
|--------------------|--------------------------|
| «Запусти проект и покажи лендинги» | `pnpm install` (один раз) + `pnpm dev` → `localhost:3000` |
| «Собери ТЗ из материалов» | скилл `/kaiten-intake` → `/intake/<slug>` |
| «Собери лендинг по ТЗ» | скилл `/kaiten-generate` → авто-роутинг → `/landings/<slug>` |
| «Проверь / что не так» | скилл `/kaiten-review`, валидаторы (гейты) |
| «Подготовь к передаче» | `handoff <slug> --require-approved`, `/approve/<slug>` |
| «Сохрани в main» | `git add/commit/push` |
| «Задеплой на Vercel» | `vercel` CLI / подключение репозитория к Vercel-проекту |
| «Подними справочник конвейера» | `pnpm run pipeline:docs` (поднимает dev + открывает браузер) → `localhost:3000/pipeline` |

Этапы конвейера соответствуют фазам пайплайна P0–P8 (`docs/pipeline.md`):
Разбор ≈ P0–P1, Макет ≈ P2, Блоки ≈ P3+P5+P8, Тексты ≈ P6, Проверки ≈ P7 + валидаторы.

⚠️ **До воркшопа проверьте сами:** что репозиторий разворачивается на Vercel
(monorepo: корень приложения — `apps/web`), и что у всех участников есть доступ
к репо. Деплой-шаг воркшопа должен быть отрепетирован.
