Главное правило работы

Вы просто говорите по-русски

Никаких команд, терминала и кода руками. Вы открываете проект в Claude Code и пишете в чат обычными словами — что нужно. Агент сам делает всю техническую часть.

«Запусти проект и покажи лендинги в браузере»
«Собери лендинг по моим материалам»
«Здесь не та картинка — поставь подходящую и покажи, как стало»

Терминологию вы будете узнавать по ходу: «локальный сервер», «генерация», «коммит в main», «деплой» — это слова из ответов агента. Знать их полезно, печатать команды — не нужно.

Шаг 1–2 · основа

GitHub — это Word с версиями

Как Word, только с историей

Всегда видно, кто и что менял, и всегда можно вернуться назад. Ничего не теряется и не ломается «насовсем».

…и внутри лежит код

Код — это правила и блоки, по которым нейронка собирает результат. Меняете правила → меняется результат. Этим вы и управляете.

Изменилисказали агенту, что поправить
→
Сохранили в mainизменение попало в общий проект
→
Проверилипосмотрели результат в браузере
→
Видно всемкоманда работает с новой версией
Почему мы работаем в одной ветке main: ветки нужны большим командам разработчиков, чтобы не мешать друг другу. Нам важнее скорость и простота — одна общая версия, история всё страхует.
Шаг 3 · как собирается лендинг

Конвейер: запрос на входе — лендинг на выходе

Репозиторий — это последовательный пайплайн. Всё, что между входом и выходом, — шаги, которые выполняются строго по порядку. У каждого шага — свои правила, и они лежат тут же, в репозитории.

Шаг 4 · чиним «не очень»

Плохой результат — это диагноз, а не приговор

Каждый вид брака указывает на конкретный шаг конвейера. Вы не переделываете лендинг руками — вы чините правило того шага, который ошибся. Один раз починили — все следующие лендинги собираются правильно.

Конвейер сам подсвечивает брак — «красные блоки» в отчёте проверок. Увидели → поняли шаг → починили правило → перегенерировали.
Шпаргалка

Универсальный промпт для правок

Один шаблон на все случаи. Подставляете своё в квадратные скобки.

  1. Вот скриншот [приложить картинку].
  2. Проблема: [что не так — конкретно].
  3. Как должно быть: [желаемый результат].
  4. Поменяй правило на шаге [название шага], чтобы такого больше не было.
  5. Перегенерируй и покажи, как стало.
  6. Если ок — сохрани изменения в репозиторий (запушь в main).
Пример. «Вот скриншот. Проблема: в разделе с возможностями стоит доска задач, а это лендинг про CRM. Как должно быть: воронка продаж или карточка клиента. Поменяй правило на шаге ‘Блоки’, чтобы для CRM не подставлялись блоки из проектного управления. Перегенерируй и покажи. Если ок — сохрани в main.»

Домашка · часть 2 · вместе с разработкой

Правила передачи в разработку — в конвейер

У разработчиков есть требования, каким должен быть лендинг, чтобы его приняли без переделок. Сейчас эти требования живут у них в головах. Задача — добыть их и встроить в конвейер, чтобы каждый готовил лендинг сразу по правилам.

1Сходите к разработчикампопросите Markdown-документ: технические требования к передаче лендинга
→
2Отдайте агенту«добавь эти правила в репозиторий как правила передачи в разработку»
→
3Правила в конвейереэтап «Передача в разработку» теперь проверяет по ним
→
4Пакуете промптомготовый пакет — без переделок и вопросов
«Лендинг готов — упакуй его для команды разработки по правилам передачи из репозитория»
Зачем: разработка получает предсказуемый пакет, который не нужно пересобирать руками. А правила, однажды добавленные в репозиторий, работают для всех — это и есть развитие конвейера.

Роудмеп · следующая большая задача · нужен ответственный

Конвейер статей: цель — ×2 выпускаемых статей

Скилл для статей у команды уже есть, но нет автоматизированного конвейера — как у лендингов. Строить его будем на следующих этапах. Сейчас нужен research: один человек берёт его в работу и готовит всё, чтобы конвейер можно было собрать.

1Опишите бизнес-процесскак статьи рождаются сейчас: от идеи до публикации, где узкие места
→
2Исследуйте рынокоткрытые репозитории с готовыми конвейерами создания контента для сайтов
→
3Соберите лучшие практикинесколько источников: что реально работает лучше всего
→
4Подготовьте материалывсё, что нужно, чтобы собрать свой конвейер статей
→
Конвейер статейстроим на следующем этапе · выпуск ×2
«Проведи research: лучшие открытые репозитории с конвейерами создания контента — сравни подходы и собери лучшие практики»
Критерий успеха задачи: после внедрения конвейера команда выпускает статей в два раза больше. Research — первый шаг; кто берёт — договариваемся на разборе.
